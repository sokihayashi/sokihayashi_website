## Works CMS設計案 ★最初の本丸

### いま何が起きているか
- `Works.mzh41.js` には既に**タグ絞り込みの土台**があり、「Works本体」+「カテゴリ」の
  2つのコレクションを前提にした作りです(下記)。
- ただしフィールドキーが仮置き(`'number'` など)で、**実際のコレクション構造と未接続**です。

```js
const CATEGORIES_COLLECTION = 'lcks7c0hpd'; // 仮のコレクションID
const WORKS_CATEGORIES_FIELD_KEY = 'number'; // 仮のフィールドキー
```

### ゴール
「20点超のサムネを手で並べる」のをやめ、**1作品=1行の表(CMSコレクション)で管理 → 画面は自動生成**。
作品を足す/直すは、Wixの **コンテンツ マネージャー**(Excelのような管理画面)に行を足すだけ。

### コレクション設計(林さんがエディタで作成する部分)

**① コレクション名:`Works`(作品本体)**

| フィールド表示名 | フィールドキー(英数) | 型 | 用途 |
|---|---|---|---|
| 作品名 | `title` | テキスト | 案件名 |
| サムネイル | `thumbnail` | 画像 | 一覧の画像 |
| クライアント | `client` | テキスト | クライアント名 |
| 担当範囲 | `role` | テキスト | 例:ロゴ, VI, 撮影 |
| 年 | `year` | 数値 or テキスト | 2024 等 |
| 外部リンク | `externalUrl` | URL | 詳細・事例ページ |
| 説明 | `description` | リッチテキスト | キャプション |
| カテゴリ | `categories` | **複数参照**(→ `Categories`) | 絞り込み用 |
| 並び順 | `order` | 数値 | 表示順の制御(任意) |

**② コレクション名:`Categories`(分類)**

| フィールド表示名 | フィールドキー | 型 | 用途 |
|---|---|---|---|
| 名称 | `title` | テキスト | 例:ブランディング |
| スラッグ | `slug` | テキスト | 例:`branding`(絞り込みボタンの値) |

> 💡 既存コードは `slug` で絞り込む設計なので、`Categories` 側に **`slug`** を必ず作るのが肝。

### 分類軸の決め方(林さんの判断ポイント)
引き継ぎ資料の「領域別 / 年代別 / 業態別」のうち、おすすめは **領域別**。
プロフィールの肩書きと一致させると一貫性が出ます:

`branding` / `logo-vi` / `graphic` / `web` / `ui` / `movie` / `photo` / `3dcg` / `sculpture`

→ ここは林さんの作品ラインナップを見て決める領域。決まったら教えてください。

### 手順の流れ(分業)
1. **林さん**:エディタで上記2コレクションを作成 → 作品データを数件入力 → ページに
   リピーター(Repeater)を置き、Works コレクションにデータ接続(データバインディング)。
2. **林さん**:絞り込み用のボタン/セレクタを置き、イベントを `selectTags1_change` に接続。
3. **林さん**:実際のフィールドキーとコレクションIDを私に共有(コンテンツマネージャーの
   「設定」で確認できます)。
4. **Claude**:`Works.mzh41.js` を実構造に合わせて確定・整理。並び順や複数選択にも対応。

> ⚠️ いまの `Works.mzh41.js` は仮の状態。**実コレクションができてキーが分かるまで、
> コードは触らず温存します**(今いじると、半分できている設定を壊すため)。
